# Этап 3. Глубокий анализ: РС «Обороты»

Дата анализа: 2026-07-18  
Статус: **реализовано** — регистр удалён, запись из быстрого ввода убрана.

---

## 1. Назначение регистра (по метаданным)

`РегистрСведений.Обороты` — периодический до **секунды** регистр сведений.

| Часть | Состав |
|-------|--------|
| Измерения | `ДатаДокумента`, `ВидОперации`, `Категория` (доходы \| расходы), `Счет`, `Валюта` |
| Ресурсы | `СуммаДокумента`, `СуммаПрихода`, `СуммаРасхода` |
| Период записи | стандартный `Период` (секунды) |

Форм, модулей, макетов у регистра **нет** — только `Обороты.mdo`.

Включён в подсистему `РегистрыСведений` и в корень конфигурации.

---

## 2. Кто пишет

Единственный писатель:

`ОбщийМодульСервер.ДобавитьДанныеВРСОбороты` ← вызывается из `СоздатьИПровестиОперацию` (быстрый ввод).

```
СоздатьИПровестиОперацию
  ├─ СоздатьДокументДвижения  → документ + РН Доходы/Расходы + РН Баланс
  └─ ДобавитьДанныеВРСОбороты → РС Обороты   ← параллельный контур
```

Особенности записи:
- `Период = ТекущаяДата()` (момент записи), а не дата операции;
- `ДатаДокумента` дублируется отдельным измерением;
- при расходе `СуммаДокумента` пишется **отрицательной**.

Документы `Доходы` / `Расход` через `ОбработкаПроведения` в РС **не пишут**.

Итог: операция через форму документа и через быстрый ввод дают **разное** состояние РС.

---

## 3. Кто читает

Поиск по всей `src/` на `РегистрСведений.Обороты` / использование РС:

| Место | Обращение к РС Обороты |
|-------|-------------------------|
| Начальный экран | **нет** (использует `РегистрНакопления.*.Обороты` — виртуальная таблица РН) |
| ОтчетПоРасходам | **нет** → `РегистрНакопления.Расходы.Обороты` |
| ОтчетПоДоходам | **нет** → `РегистрНакопления.Доходы.Обороты` |
| ОтчетПоВзаиморасчетам | **нет** → `РегистрНакопления.Баланс.ОстаткиИОбороты` |
| Обмен `Мобильные` | состав плана пуст; явных ссылок на РС нет |
| Прочий BSL | **нет чтений** |

Вывод: регистр **никто не читает**. Это мёртвый параллельный контур.

Имя «Обороты» в запросах начального экрана — это **виртуальная таблица регистра накопления**, не РС.

---

## 4. Сравнение с действующей моделью учёта

Рабочая модель (используется UI и отчётами):

```
Документ Доходы/Расход
  → РН Доходы / РН Расходы (оборотные, по категориям и валюте)
  → РН Баланс (остатки по счёту и валюте)
```

РС Обороты пытается смешать в одной записи:
- вид операции;
- категорию;
- счёт;
- приход и расход.

Это хуже, чем уже существующие РН:
- нет связи с регистратором (документ);
- нет автоматической очистки при отмене проведения / удалении;
- нет использования в отчётах.

---

## 5. Дефекты и риски регистра

| ID | Severity | Проблема |
|----|----------|----------|
| E3-01 | Критично | Два канала ввода → разное наполнение РС; аналитика по РС была бы ложной |
| E3-02 | Критично | Нет потребителей: хранение без пользы, усложнение транзакции быстрого ввода |
| E3-03 | Важно | Ключ без уникального ID операции: при совпадении измерений + периода (секунда) — перезапись `Записать()` |
| E3-04 | Важно | `Период = ТекущаяДата()`, а измерение `ДатаДокумента` — путаница периодов отбора |
| E3-05 | Важно | При отмене проведения / удалении документа строка РС **остаётся** |
| E3-06 | Желательно | Дублирует смысл РН; поддержка двух контуров дороже |

Пункт плана 3.4 («уникальный ключ») имеет смысл **только если** регистр оставляем. При удалении — не нужен.

---

## 6. Варианты решения

### Вариант A — удалить РС «Обороты» (рекомендуется)

1. Убрать вызов `ДобавитьДанныеВРСОбороты` из `СоздатьИПровестиОперацию`.
2. Удалить процедуру `ДобавитьДанныеВРСОбороты`.
3. Упростить транзакцию: достаточно `СоздатьДокументДвижения` (проведение документа уже транзакционно).
4. Удалить объект метаданных `InformationRegisters/Обороты`.
5. Убрать из `Configuration.mdo` и подсистемы `РегистрыСведений`.
6. Подсистему `РегистрыСведений` — оставить пустой или удалить (сейчас в ней только этот регистр).

Плюсы: один источник истины, проще код, нет рассинхрона.  
Минусы: в существующих ИБ останутся «осиротевшие» данные РС (не мешают, если объект удалён из конфигурации — при обновлении данные регистра обычно очищаются платформой).

### Вариант B — заполнять только из проведения документов

Потребовалось бы:
- подписки / код в `ОбработкаПроведения` обоих документов;
- очистка при отмене проведения;
- уникальный ключ (регистратор или UUID);
- **и всё равно** нужен хоть один читатель (отчёт/экран), иначе бессмысленно.

Плюсы: теоретическая «единая запись».  
Минусы: дублирование РН без выгоды; усложнение проведения.

### Вариант C — оставить как есть

Не рекомендуется: уже подтверждённый технический долг.

---

## 7. Рекомендация

**Принять вариант A: удалить регистр сведений «Обороты».**

Обоснование:
- ноль чтений в конфигурации;
- полный функциональный дубль РН для UI/отчётов;
- после этапа 1 быстрый ввод и так создаёт документы с корректным проведением.

После удаления пункты 3.3–3.4 плана закрываются так:
- 3.3 — убрать запись (обязательно);
- 3.4 — **не применимо** (регистр удалён).

---

## 8. План работ после утверждения

1. Упростить `СоздатьИПровестиОперацию` (без РС, при необходимости без лишней внешней транзакции).
2. Удалить `ДобавитьДанныеВРСОбороты`.
3. Удалить метаданные РС и ссылки в конфигурации/подсистеме.
4. Проверить, что отчёты и начальный экран не затронуты (они не используют РС).
5. Обновить чек-лист этапа 3.

---

## 9. Вне объёма этапа 3

- Состав плана обмена `Мобильные` — этап 5.
- Реквизит `Счета.Баланс` — этап 6.
