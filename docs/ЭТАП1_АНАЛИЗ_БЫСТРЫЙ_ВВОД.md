# Этап 1. Глубокий анализ: быстрый ввод и проведение

Дата анализа: 2026-07-18  
Статус: анализ завершён, правки кода ещё не начаты.

---

## 1. Точка входа и поток данных

```
НачальныйЭкран
  └─ ДобавитьРасход / ДобавитьДоход
       └─ ОткрытьФорму(ОбщаяФормаДокумента, ВидОперации)
            └─ Записать (команда формы)
                 └─ ОбщийМодульКлиент.ВыполнитьДвиженияВФоне(Массив)
                      └─ ОбщийМодульСервер.ВыполнитьДвиженияВФоне
                           └─ ФоновыеЗадания.Выполнить("…ЗаписатьДанные", Массив)
                                └─ ЗаписатьДанные
                                     ├─ СоздатьДокументДвижения → Документ.Записать(Проведение)
                                     │    └─ ОбработкаПроведения → РН Доходы/Расходы + Баланс
                                     └─ ДобавитьДанныеВРСОбороты → РС Обороты (параллельно)
            └─ Закрыть()  ← сразу, не дожидаясь фона
                 └─ ПослеЗакрытияФормыДокумента → ПодготовитьФормуНаСервере()
```

Альтернативный путь (стандартные формы документов `Доходы` / `Расход`):
- пользователь заполняет форму документа вручную;
- проведение только через `ОбработкаПроведения`;
- **РС `Обороты` не заполняется**.

Итог: два канала ввода дают **разное состояние данных**.

## 3. Что происходит по шагам

### 3.1. Открытие формы

- Параметр `ВидОперации` приходит с начального экрана.
- `Валюта` подставляется из константы `ВалютаУчета` (может быть пустой после кривого первого запуска — этап 2).
- `ДатаДокумента = ТекущаяДата()`.
- Поля формы создаются **программно** (`ДобавитьРеквизитНаФорму`): дата, счёт, категория, валюта, сумма + кнопка «Записать».
- Реквизиты формы уже описаны в `Form.form` (ВидОперации, ДатаДокумента, Валюта, Сумма, Счет, КатегорияДоходов, КатегорияРасходов).
- У полей формы нет `АвтоОтметкаНезаполненного` / проверки заполнения.

### 3.2. Команда «Записать»

```bsl
МассивДанных = [ДатаДокумента, ВидОперации, Категория, Счет, Валюта, Сумма]
ОбщийМодульКлиент.ВыполнитьДвиженияВФоне(МассивДанных);
Закрыть();
```

Проблемы:
1. **Нет валидации** до вызова.
2. Категория выбирается эвристикой `Если ЗначениеЗаполнено(КатегорияДоходов)` — при пустых обоих в массив уйдёт пустая `КатегорияРасходов`.
3. Форма **закрывается сразу** после постановки фонового задания.

### 3.3. Фоновое задание

```bsl
ФоновыеЗадания.Выполнить("ОбщийМодульСервер.ЗаписатьДанные", МассивДанных, Новый УникальныйИдентификатор, "…");
```

- Передача массива параметров в метод с 6 аргументами — **корректный** контракт платформы.
- Для короткой операции «создать и провести один документ» фон **не нужен**: усложняет UX, отладку и обработку ошибок.
- На мобильной платформе фон есть, но результат задания **никто не ждёт и не проверяет**.

### 3.4. `ЗаписатьДанные` — критичный дефект

```bsl
НачатьТранзакцию();
Попытка
    СоздатьДокументДвижения(...);
    ДобавитьДанныеВРСОбороты(...);
Исключение
    ОтменитьТранзакцию();
    // ошибка проглочена — пользователю ничего
КонецПопытки;
Если ТранзакцияАктивна() Тогда
    ЗафиксироватьТранзакцию();
КонецЕсли;
```

Последствия:
- Ошибка проведения / записи / РС → откат, **тишина**.
- Форма уже закрыта → пользователь уверен, что всё сохранено.
- `Сообщить("Запись успешно добавлена")` внутри `ДобавитьДанныеВРСОбороты` в фоне почти бесполезен и противоречит правилам проекта.

### 3.5. Создание документа

- Создаётся объект `Документы.Доходы` или `Документы.Расход`, заполняются реквизиты, `Записать(РежимЗаписиДокумента.Проведение)`.
- Проверка заполнения категории сработает только если платформа её выполнит при записи; **счёт, валюта, сумма** в метаданных без `fillChecking`.
- В модулях объектов **нет** `ПередЗаписью` / проверок суммы / заполненности счёта.

### 3.6. Обновление начального экрана

`ПослеЗакрытияФормыДокумента` сразу вызывает `ПодготовитьФормуНаСервере()`.

Гонка:
- фон ещё не успел провести документ → сводка **без новой операции**;
- пользователь видит «старые» цифры и может ввести операцию повторно;
- либо наоборот — думает, что запись «не сработала».

## 4. Сравнение с записью через форму документа

| Аспект | Быстрый ввод | Форма документа |
|--------|--------------|-----------------|
| Валидация UI | нет | частичная (только категория в метаданных) |
| Проведение | да | да |
| РС Обороты | да | нет |
| Обратная связь об ошибке | нет | стандартные сообщения платформы |
| Обновление дашборда | сразу (часто рано) | вручную / при возврате |

Проведение в обоих случаях по сути корректное (вид движения Баланса верный). Проблемы этапа 1 — **обвязка**, не формулы движений.

## 6. Целевое поведение (после этапа 1)

1. Пользователь нажимает «Записать».
2. Клиент проверяет: дата, вид операции, категория по виду, счёт, валюта, сумма > 0.
3. Сервер повторяет те же проверки.
4. Синхронно создаётся и проводится документ (без фона для одной операции).
5. При ошибке — сообщение пользователю, форма **остаётся открытой**.
6. При успехе — форма закрывается с результатом; начальный экран обновляется уже по факту записи.
7. Проведение документов через стандартные формы также защищено проверками заполнения / суммы.

Запись в РС `Обороты` на этапе 1 **не расширяем**; решение по регистру — этап 3.

## 7. Предлагаемый объём правок (когда перейдём к коду)

1. **`ОбщийМодульСервер`**
   - заменить `ВыполнитьДвиженияВФоне` на синхронный экспорт, например `СоздатьИПровестиОперацию`;
   - `ЗаписатьДанные`: убрать глушение исключений; при ошибке — `ВызватьИсключение` / вернуть описание;
   - вынести `ПроверитьДанныеОперации(...)`.

2. **`ОбщийМодульКлиент`**
   - вызывать синхронный серверный метод; пробрасывать ошибку на форму.

3. **`ОбщаяФормаДокумента`**
   - валидация перед записью;
   - закрывать только при успехе;
   - при желании возвращать `Истина` в `Закрыть(Результат)` для явного сигнала начальному экрану.

4. **Документы `Доходы` / `Расход`**
   - `fillChecking` для `Счет`, `Валюта`, `Сумма`;
   - в `ПередЗаписью` или `ОбработкаПроведения`: отказ при `Сумма <= 0` и незаполненных ключевых реквизитах.

5. **Начальный экран**
   - обновлять сводку только если результат закрытия = успех (или всегда после закрытия, но уже после синхронной записи — гонки не будет).

## 8. Риски и ограничения

- MCP `docsearch` / `syntaxcheck` на момент анализа недоступны — при правках проверить синтаксис, как только сервер поднимется.
- Проведение документа внутри явной транзакции в `ЗаписатьДанные` допустимо, но после отказа от фона можно упростить: одна запись документа с проведением без внешней обёртки (платформа сама ведёт транзакцию записи), если РС `Обороты` уберём на этапе 3.
- Точность суммы: форма `Число(15,2)`, документ `Число(10,2)` — выровнять при правках метаданных.

## 9. Вывод

Архитектура «документ → регистры накопления» для учёта **нормальная**.  
Сломлен контур **быстрого ввода**: асинхронность без обратной связи + подавление ошибок + отсутствие валидации + раннее обновление UI.

Этап 1 должен сделать быстрый ввод **предсказуемым и безопасным** (синхронно, с проверками, с понятной ошибкой). Параллельный РС `Обороты` трогать минимально — полное решение на этапе 3.
